## Verdict
BLOCK

Solid layering and test discipline for a same-day scope, but the two-phase `signUp` + `insert` in the repository has no compensating action on partial failure, which can permanently lock a real email address out of registration with no recovery path — that's a correctness bug in the one thing this feature is supposed to guarantee.

## Summary
This is a clean, well-scoped slice: a proper port/adapter (`TeacherRepository` / `SupabaseTeacherRepository`), typed errors, a shared zod schema reused client- and server-side, and a straightforward controlled form with sensible accessibility (labeled inputs, `aria-invalid`/`aria-describedby`, `role="alert"`/`role="status"`, disabled-while-submitting button). Test coverage on the schema and the adapter's error-mapping is good and matches CLAUDE.md's "light unit tests on core business logic" bar. The three issues that matter: (1) `SupabaseTeacherRepository.register` creates the auth user first and the `teachers` row second with no rollback — any insert failure (RLS misconfig, transient DB error, network blip) leaves an orphaned auth user and permanently reserves that email, since a retry will always hit `signUp`'s duplicate-email path first with no way to complete or recover; (2) every server-side failure is swallowed into a generic 500 with zero logging, so the exact failure in (1) would be undebuggable in production; (3) the success copy ("You can now sign in") is factually wrong — the user already has an active session from the `signUp` call that ran inside the route handler.

## Findings

### 🔴 Blockers
- [ ] **ms-next-app/lib/teachers/supabase-repository.ts:28-60** — `register()` calls `auth.signUp()` then `insert()` into `teachers` as two independent, non-transactional steps with no compensation. If the insert fails for *any* reason (RLS policy misconfigured, transient Postgres error, network blip — all plausible in a fresh Supabase project), the auth user from `signUp` is left orphaned. Every subsequent registration attempt with that email will now fail at the `signUp` step with `DuplicateEmailError` (line 34-36) — the user is told "an account already exists" forever, but there is no `teachers` row and no login page to recover the account. This is not a hypothetical: it's the exact failure mode the plan's own Risks section calls out ("RLS blocks the insert... every registration will 500") but the plan only anticipated a one-time 500, not a permanently unrecoverable email. **Fix:** on insert failure, compensate — e.g. use a service-role admin client to call `auth.admin.deleteUser(user.id)` before rethrowing, so the email becomes available again; or better, do the insert via a Postgres function (RPC) invoked from `signUp`'s trigger/webhook so the two writes are atomic from the caller's perspective. At minimum, don't leave the system in a state a legitimate user can't get out of.
- [ ] **ms-next-app/app/v1/register/route.ts:33-47** — Every failure path (`RepositoryError` and the unexpected-error fallback) returns a generic 500 with zero server-side logging (no `console.error`, no logger call). The original error and its `cause` (the actual Postgres/GoTrue failure detail) are discarded entirely. Combined with the blocker above, this means an orphaned-user incident in production leaves no trace anywhere to diagnose it — the operator has only "something failed for someone, at some point." **Fix:** log the error (with cause) server-side before returning the generic response, e.g. `console.error("register failed", error)` at minimum, or wire the lightweight logger the rest of the app will eventually use. This is not "add pino now" — it's "don't return 500 into a void."

### ⚠️ Should fix
- [ ] **ms-next-app/app/v1/register/route.ts:37-42** — The `error instanceof RepositoryError` branch and the final fallback branch (line 43-46) return byte-for-byte identical responses. This isn't wrong, but it reads as if `RepositoryError` gets special handling when it doesn't, and it hides the fact that any *other* unexpected error type is silently treated the same as a known repository failure. **Fix:** either give `RepositoryError` a distinct response (unlikely needed here) or collapse the two branches into one `else` and log the error type, so it's clear the fallback is intentional and not an oversight.
- [ ] **ms-next-app/app/v1/register/route.ts** (whole file) — no unit test exercises the route handler's status-code mapping (201 / 400 / 409 / 500). Task 2's acceptance criteria deferred this to a manual `curl` that the plan itself says was never run (sandbox restriction, see plan Assumptions). As written, the 400/409/500 mapping in this file is genuinely untested — only the schema and the repository are tested in isolation, and nothing verifies they're wired together correctly (e.g. that `DuplicateEmailError` really does produce a 409, not some other status). **Fix:** add a light unit test that mocks `SupabaseTeacherRepository.register` (or injects a repository double) and asserts the four status codes, per CLAUDE.md's "light unit tests... focused on core business logic" — this route's error mapping is exactly that.
- [ ] **ms-next-app/app/register/RegisterForm.tsx:41-42** — Success copy reads "You can now sign in and finish setting up your profile," but the user already has an active session at this point — the `signUp()` call happened server-side inside the route handler using the cookie-based server client, and the resulting session cookie is already set on the response by the time the client sees `201`. Telling the user to "sign in" when they're already authenticated is confusing and will read as broken once a login/dashboard page exists and they land on it already logged in. **Fix:** rephrase to something like "You're registered and signed in. Head to your dashboard to finish your profile" (once that page exists), or at minimum drop "you can now sign in."

### 💭 Nits
- [ ] **ms-next-app/lib/supabase/server.ts** — no `import "server-only"` guard. Nothing currently imports this module from client code, but it holds cookie-based auth logic that must never end up in a client bundle; a one-line `import "server-only"` at the top turns a future accidental client import into a build error instead of a runtime surprise.
- [ ] **ms-next-app/lib/teachers/supabase-repository.test.ts** — no test covers the "hourlyPrice omitted" path at the adapter level (only the schema test covers omission — see `register-schema.test.ts:18-25`). The adapter test fixture always passes `hourlyPrice: 50`, so `hourly_price: hourlyPrice ?? null` on `supabase-repository.ts:50` is never exercised with `undefined`. Cheap to add: one more `createMockSupabase` case without `hourlyPrice` in the input.
- [ ] **ms-next-app/app/v1/register/route.ts:33-47** — `DuplicateEmailError`/`RepositoryError` are only distinguished by `instanceof`; fine here, but worth a comment noting *why* both errors get individually checked (readability for the next route that copies this pattern for `/v1/booking` or `/v1/checkout`).

### 💡 Suggestions
- [ ] Once a second route handler exists (`/v1/booking`, `/v1/checkout`, `PATCH /v1/teachers`), extract the try/catch + typed-error-to-status-code mapping in `route.ts:30-47` into a shared helper (e.g. `mapErrorToResponse`) so each route isn't reimplementing the same three-way branch. Not worth doing for a single route today — flagged for follow-up only.
- [ ] Add a `.env.example` in `ms-next-app/` documenting `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`. The plan's own Assumptions section notes these var names were guessed and never verified against the real `.env` — a checked-in example file would make the requirement explicit and catch a mismatch at `npm install` time for the next person, instead of at first-run `throw new Error("Missing required environment variable...")`.
- [ ] Consider whether `SupabaseTeacherRepository.register` should check for an existing session before calling `signUp()` at all — if the "orphaned auth user, no teachers row" state ever occurs (blocker above), a self-service recovery flow (detect "you're signed in but have no profile" and let the user complete the `teachers` insert) would be a more robust fix than a delete-and-retry compensation. Bigger than this task; noted for whoever builds the login/dashboard flow.

## Plan critique
The plan's Risks section correctly identifies "RLS blocks the insert" and "email confirmation still enabled" as risks, but frames both purely as *pre-launch verification* items ("the first real manual test... will [catch this]"), not as a *code-level* gap. That framing is why Task 1's Step 4 ("Implement `SupabaseTeacherRepository`... Map a duplicate-email response... and a unique-violation from the insert to a typed `DuplicateEmailError`") stopped at error *mapping* and never asked for error *recovery* — the plan never posed the question "what happens to the auth user we already created if the second write fails?" That's a missing invariant in the plan, not just an executor oversight: a two-phase write across two systems (GoTrue + Postgres) needs a stated compensation strategy before Task 1 is "done," and the plan's Acceptance for Task 1 (`npm run test` passes, `tsc` passes, no `app/` import) never tests for it because the risk was never named as a code requirement.
